# TOCS_MASTER_SPEC_v4.5

## 0. 문서 목적

본 문서는 TOCS(Trading Operation Control System)의 최상위 기준 문서다.

TOCS의 모든 기획, 설계, 개발, AI 프롬프트, QA 검토는 본 문서를 기준으로 한다.

---

# 1. TOCS 핵심 정의

TOCS는 Formula First Architecture 기반의 Trading Operation Control System이다.

TOCS는 단순 거래관리, ERP, CRM, 그룹웨어가 아니다.

TOCS는 Formula를 중심으로 돈 흐름, 세무 흐름, 물류 흐름, 정산 흐름, 수익 흐름을 통제하는 현금흐름 중심 시스템이다.

핵심 데이터 흐름은 아래 구조를 따른다.

```text
Formula
↓
Derived Data
↓
Accumulated / Aggregated Data
↓
Dashboard
↓
KPI / Management Indicators
```

모든 KPI와 경영지표는 Formula에서 파생된다.

---

# 2. Formula 정의

Formula는 TOCS의 최상위 핵심 단위이자 운영 원장(Source of Truth)이다.

Formula는 아래 흐름을 하나로 묶는다.

- 돈 흐름
- 세무 흐름
- 물류 흐름
- 정산 흐름
- 수익 흐름

Formula 구조를 변경하면 파생 데이터, 누적 집계, 대시보드, KPI가 모두 영향을 받는다.

따라서 Formula 구조 변경은 최고 수준 검토 대상이다.

---

# 3. Formula 기본 구조 원칙

## 3.1 Formula 1개 = 품목 1개

V1 기준 Formula 1개에는 품목 1개만 연결한다.

다품목 거래는 품목별 Formula를 별도로 생성한다.

따라서 V1에서는 `formula_items` 같은 다품목 하위 테이블을 생성하지 않는다.

## 3.2 Deal / Order 중심 구조 금지

TOCS는 Deal 중심 시스템이 아니다.

아래와 같은 최상위 Entity는 생성하지 않는다.

- Deal
- Order
- Project
- Pipeline
- Campaign

Formula가 원장이며 모든 하위 데이터는 Formula에 연결한다.

# 6. Formula Participants / 거래 라인 구조

Formula 내 A > B > C > D 구조는 `formula_participants` 라인 구조로 저장한다.

`sequence_order`로 포뮬러 흐름을 표현한다.

예시:

```text
1 CJ제일제당: buy 0 / sell 710
2 지오웍스: buy 710 / sell 1010
3 네이처인사이트: buy 1010 / sell 1200
4 에코앤리사이클: buy 1200 / sell 0
```

중요 규칙:

- 매입가 0 허용
- 매출가 0 허용
- 시작점은 매입가 0 가능
- 종료점은 매출가 0 가능
- Company는 고정 역할을 갖지 않고 Formula 내부에서 역할이 결정됨
- 참여자 순서 변경, 중간 회사 추가/삭제/변경 시 버전 또는 스냅샷을 남김

# 7. 입출금 엔진

TOCS KPI는 실제 은행 입출금 기준이다.

확정매출은 거래 생성 기준이나 계산서 기준이 아니라 실제 은행 입금액 기준으로 계산한다.

## 7.1 입출금 확인 방식

입금/출금은 실무자가 실제 법인통장 입출금 내역 또는 대표이사 입출금 문자를 확인한 후 완료 처리한다.

자동 확정이 아니라 실무 확인 기반 버튼 처리 구조를 기본으로 한다.

## 7.2 입금 정보

- 입금구분
- 입금금액
- 입금예정일
- 입금상태
- 실입금일
- 입금회사

## 7.3 출금 정보

- 출금구분
- 출금금액
- 출금예정일
- 출금상태
- 실출금일
- 출금회사

## 7.4 완료 / 완료취소

입금완료 및 출금완료 후에도 완료취소가 가능해야 한다.

사람이 처리하는 업무이므로 실수, 오입력, 변경 가능성을 허용한다.

단, 완료 처리와 완료취소는 반드시 이력으로 남겨야 한다.

삭제하지 않고 취소 상태로 보존한다.

## 7.5 다건 입출금

입출금 엔진은 다건 구조를 전제로 설계한다.

입금 구조:

```text
입금예정금액
↓
입금내역(다건)
↓
누적입금
↓
입금률
↓
미수금
```

출금 구조:

```text
출금예정금액
↓
출금내역(다건)
↓
누적출금
↓
출금률
↓
미지급금
```

실제 현장 기준:

- 우리가 지급하는 돈은 선입금 비중이 높음
- 우리가 받는 돈은 분할입금 비중이 존재함
- 선입금 약 70%, 분할입금 약 30% 기준으로 고려
- 미수금 관리가 매우 중요함

# 8. 운송 엔진

실물 흐름과 세무 흐름은 다를 수 있다.

예시:

```text
세무 흐름:
A > B > C > D > E

실물 흐름:
A -----------------> E
```

## 8.1 운송 기본 정보

- 출고지
- 입고지
- 운송사
- 운송료 부담주체
- 운송품목
- 운송수량
- 차량수
- 총운송비
- 운송예정일

## 8.2 차량별 정보

- 차량번호
- 기사이름
- 기사연락처
- 차량별 운송비
- 차량별 운송상태
- 차량별 정산상태

## 8.3 운송비 처리 원칙

운송비가 매입가 또는 판매가에 포함되어 있어도 실제 지급 주체는 반드시 존재한다.

운송비는 Formula 내에서 별도로 관리되어야 한다.

운송비 표시 방식은 아래 유형을 지원한다.

- 매입가 포함
- 판매가 포함
- 별도 비용처리

## 8.4 운송비 지급 대상

운송비 지급 대상은 세금계산서를 발행한 운송사업체다.

실제 운송 주체, 기사, 차주와 세무상 정산 대상은 다를 수 있다.

따라서 운송 엔진은 실물 운송 주체와 세무 정산 주체를 분리 관리한다.

중요 원칙:

```text
carrier_company_id = 세금계산서를 발행한 운송사업체
운송비 지급 대상 = 세금계산서를 발행한 운송사업체
```

따라서 고정 공식보다 직접입력과 차등배분을 우선 지원해야 한다.

- 직접입력

## 9.2 셰어 방식

2. 정액

## 9.3 셰어 정보

# 10. 계산 엔진 v1

## 10.1 기본 이익 계산 공식

```text
순이익 = 총매출 - 총매입 - 비용 - 셰어
```

## 10.2 계산 항목

- 총매입 = 수량 × 매입단가
- 총매출 = 수량 × 매출단가
- 비용 = 운송비 + 수수료 + 기타 직접비용
- 셰어 = 직접입력 또는 계산 방식에 따른 배분금액
- 순이익 = 총매출 - 총매입 - 비용 - 셰어
- 이익률 = 순이익 / 총매출

## 10.3 부가세 기준

TOCS의 기본 계산은 부가세 포함 금액을 기준으로 한다.

향후 확장 기능으로 아래 항목을 개발할 수 있다.

- 공급가
- 부가세
- 합계금액
- 예상 부가세
- 부가세 예상 납부/환급 영향

## 10.4 환율 기준

수입/수출/혼합 거래의 환율은 기본적으로 계약시 환율을 기준으로 한다.

단, 거래 진행 중 환율 변동이 발생할 수 있으므로 아래 구조를 허용한다.

- 계약환율
- 수정환율
- 환율변경 사유
- 환율변경 이력

## 10.5 계산 결과 스냅샷

Formula 수정 또는 신규 버전 생성 시 계산 결과 스냅샷을 저장해야 한다.

목적:

- 과거 데이터 훼손 방지
- 버전별 수익 비교
- 정산 근거 보존
- 세무 증빙 보존

스냅샷 예시:

- 계산시점
- 적용환율
- 총매입
- 총매출
- 운송비
- 기타비용
- 셰어금액
- 예상순이익
- 확정순이익
- 계산버전

# 11. 계산서 / 증빙 엔진

계산서 흐름은 기본적으로 입출금 흐름 및 거래 단계와 동일하게 관리한다.

예시:

```text
A.CJ제일제당 → B.지오웍스
B.지오웍스 → C.네이처인사이트
C.네이처인사이트 → D.에코앤리사이클
```

## 11.1 계산서의 성격

계산서는 거래 진행을 막는 락 장치가 아니라, Formula 종결 판단 장치다.

원칙:

```text
계산서 없어도 거래 진행 가능
계산서 없어도 입출금 가능
계산서 없으면 Formula 종결 불가
```

## 11.2 계산서 라인 자동 생성

Formula 참여자 순서 기준으로 계산서 라인을 자동 생성할 수 있다.

예:

```text
A > B > C > D

자동 계산서 라인:
A → B
B → C
C → D
```

## 11.3 계산서 금액 구조

부가세 포함 계산이 기본이지만 계산서에서는 아래 3개를 분리 저장할 수 있어야 한다.

- 공급가
- 부가세
- 합계금액

## 11.4 증빙 원칙

증빙 구조는 관계 법령과 실무 필요성에 따라 관리한다.

단, 증빙은 거래 전 필수조건이 아니다.

거래를 먼저 진행한 후 증빙을 준비할 수 있다.

따라서 UI에서 증빙 미등록 상태를 경고할 수는 있으나, 무조건 정산 승인 차단 조건으로 두면 안 된다.

권장 구조:

- 정산상태와 증빙상태 분리
- 증빙 미등록 시 경고 표시
- 예외 승인 가능
- 승인 및 예외 승인 이력 저장

# 12. Formula 상태 관리

Formula는 단일 상태만으로 판단하지 않는다.

아래 핵심 상태를 개별 관리한다.

- 거래상태
- 입금상태
- 출금상태
- 계산서상태
- 운송상태

모든 상태는 수정 가능해야 한다.

완료 처리 후 완료취소가 가능해야 한다.

단, 모든 상태 변경은 이력으로 남긴다.

상태를 단일 `status` 하나로 압축하지 않는다.

---

# 13. Formula 종결 규칙

아래 상태가 모두 완료일 때만 Formula는 종결될 수 있다.

- 거래상태 완료
- 입금상태 완료
- 출금상태 완료
- 계산서상태 완료
- 운송상태 완료

하나라도 미완료이면 종결할 수 없다.

예를 들어 거래상태, 출금상태, 계산서상태, 운송상태가 완료되었더라도 입금상태가 미완료이면 Formula는 종결되지 않는다.

미종결 Formula는 추적관리 대상이다.

대시보드와 관리 화면에서 미종결 사유를 노출해야 한다.

# 14. Formula Version

Formula 진행 중 변수나 변경사항이 발생하면 신규 버전 또는 스냅샷을 생성한다.

주요 변경 사유:

1. 입금 지연
2. 출금 지연
3. 단가 변경
4. 수량 변경
5. 바이어 변경
6. 운송 지연
7. 운송비 변경

목적:

- 최초 구조와 변경 후 구조 비교
- 변경 이력 추적
- 세무 증빙 보존
- 정산 근거 보존
- 과거 데이터 훼손 방지

# 15. KPI 엔진

TOCS KPI는 현금흐름 중심이다.

## 15.1 확정매출

```text
확정매출 = 실제 은행 입금액
```

## 15.2 확정출금

```text
확정출금 = 실제 은행 출금액
```

## 15.3 확정순이익

```text
확정순이익 = 실입금 - 실출금 - 실제비용 - 실제셰어
```

## 15.4 미수금

```text
미수금 = 입금예정금액 - 누적입금액
```

## 15.5 미지급금

```text
미지급금 = 출금예정금액 - 누적출금액
```

## 15.6 예상 KPI

예상 KPI는 Formula 기준으로 별도 관리한다.

확정 KPI와 예상 KPI를 혼합하지 않는다.

## 15.7 대표 KPI 우선순위

S급 KPI:

- 실입금 / 확정매출
- 확정순이익
- 미수금
- 미지급금

A급 KPI:

- 예상매출
- 예상순이익
- 진행중 Formula
- 종결 Formula
- 미종결 Formula
- 거래이슈 수

B급 KPI:

- 거래처별 실적
- 품목별 실적
- 운송사별 미지급금
- 월간 목표달성률
- 연간 목표달성률
- 전월대비 성장률
- 전년동월대비 성장률

# 16. 실제 거래 검증 사례

## 16.1 포뮬러

```text
A.CJ제일제당
↓
B.지오웍스
↓
C.네이처인사이트
↓
D.에코앤리사이클
```

## 16.2 주요 특징

- 부가세 포함 거래
- 선입금 구조
- 세금계산서는 단계별 발행
- 실물은 A에서 D로 직송
- 지오웍스가 운송비를 부담하고 판매가에 운송비를 포함
- 매입 없는 회사 허용
- 매출 없는 최종 회사 허용

## 16.3 설계 반영점

- 매입가 0원 가능
- 매출가 0원 가능
- 최종 도착 회사는 매출/이익 확인 불가 가능
- 운송비 부담 주체와 운송사 정산은 별도 관리
- 세무 흐름과 실물 흐름은 분리 관리

```text

 ├─ formula_invoices
 └─ formula_shares

 ├─ formula_participants
 ├─ formula_payments

 ├─ formula_invoices
 ├─ formula_shares

 ├─ formula_status_logs
 └─ audit_logs
```

## 17.3 핵심 FK 원칙

모든 하위 데이터는 `formula_id`를 기준으로 Formula에 연결한다.

`formula_participants`는 A > B > C > D 구조를 표현하는 중심 라인이다.

입출금, 계산서, 셰어는 가능한 경우 participant 기준으로도 연결한다.

운송은 Formula에 연결하되, 실물 출고/입고 회사와 세무 정산 운송사를 분리한다.

## 17.4 Claude DB Schema Prompt

Claude에게 DB Schema 초안을 요청하기 위한 전달 문서를 생성했다.

산출물:

```text
CLAUDE_DB_SCHEMA_PROMPT_v1.md
```

해당 문서는 Claude에게 PostgreSQL Schema, Prisma Schema, 관계 설명, 인덱스, 제약조건, KPI 집계 쿼리 초안을 요청하기 위한 기준 문서다.
