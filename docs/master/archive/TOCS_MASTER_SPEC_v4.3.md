# TOCS_MASTER_SPEC_v4.3

## 0. 문서 목적

본 문서는 TOCS(Trading Operation Control System)의 최상위 기준 문서다.

TOCS의 모든 기획, 설계, 개발, AI 프롬프트, QA 검토는 본 문서를 기준으로 한다.

---

# 1. TOCS 핵심 정의

TOCS는 Formula First Architecture 기반의 Trading Operation Control System이다.

TOCS는 단순 거래관리, ERP, CRM, 그룹웨어가 아니다.

TOCS는 Formula를 중심으로 돈 흐름, 세무 흐름, 물류 흐름, 정산 흐름, 수익 흐름을 통제하는 시스템이다.

핵심 데이터 흐름은 아래 구조를 따른다.

```text
Formula
↓
Derived Data
↓
Accumulated / Aggregated Data
↓
Dashboard
↓
KPI / Management Indicators
```

모든 KPI와 경영지표는 Formula에서 파생된다.

---

# 2. Formula 정의

Formula는 TOCS의 최상위 핵심 단위이자 운영 원장(Source of Truth)이다.

Formula는 아래 흐름을 하나로 묶는다.

- 돈 흐름
- 세무 흐름
- 물류 흐름
- 정산 흐름
- 수익 흐름

Formula 구조를 변경하면 파생 데이터, 누적 집계, 대시보드, KPI가 모두 영향을 받는다.

따라서 Formula 구조 변경은 최고 수준 검토 대상이다.

# 8. 계산 엔진 v1

## 8.1 기본 이익 계산 공식

```text
순이익 = 총매출 - 총매입 - 비용 - 셰어
```

## 8.2 계산 항목

- 총매입 = 수량 × 매입단가
- 총매출 = 수량 × 매출단가
- 비용 = 운송비 + 수수료 + 기타 직접비용
- 셰어 = 직접입력 또는 계산 방식에 따른 배분금액
- 순이익 = 총매출 - 총매입 - 비용 - 셰어
- 이익률 = 순이익 / 총매출

## 8.3 부가세 기준

TOCS의 기본 계산은 부가세 포함 금액을 기준으로 한다.

향후 확장 기능으로 아래 항목을 개발할 수 있다.

- 공급가
- 부가세
- 합계금액
- 예상 부가세
- 부가세 예상 납부/환급 영향

## 8.4 환율 기준

수입/수출/혼합 거래의 환율은 기본적으로 계약시 환율을 기준으로 한다.

단, 거래 진행 중 환율 변동이 발생할 수 있으므로 아래 구조를 허용한다.

- 계약환율
- 수정환율
- 환율변경 사유
- 환율변경 이력

## 8.5 계산 결과 스냅샷

Formula 수정 또는 신규 버전 생성 시 계산 결과 스냅샷을 저장해야 한다.

# 9. 계산서 / 증빙 원칙

계산서 흐름은 기본적으로 입출금 흐름 및 거래 단계와 동일하게 관리한다.

예시:

```text
A.CJ제일제당 → B.지오웍스
B.지오웍스 → C.네이처인사이트
C.네이처인사이트 → D.에코앤리사이클
```

증빙 구조는 관계 법령과 실무 필요성에 따라 관리한다.

단, 증빙은 거래 전 필수조건이 아니다.

거래를 먼저 진행한 후 증빙을 준비할 수 있다.

따라서 UI에서 증빙 미등록 상태를 경고할 수는 있으나, 무조건 정산 승인 차단 조건으로 두면 안 된다.

권장 구조:

- 정산상태와 증빙상태 분리
- 증빙 미등록 시 경고 표시
- 예외 승인 가능
- 승인 및 예외 승인 이력 저장

# 10. Formula 상태 관리

Formula는 단일 상태만으로 판단하지 않는다.

아래 핵심 상태를 개별 관리한다.

- 거래상태
- 입금상태
- 출금상태
- 계산서상태
- 운송상태

모든 상태는 수정 가능해야 한다.

완료 처리 후 완료취소가 가능해야 한다.

단, 모든 상태 변경은 이력으로 남긴다.

---

# 11. Formula 종결 규칙

아래 상태가 모두 완료일 때만 Formula는 종결될 수 있다.

- 거래상태 완료
- 입금상태 완료
- 출금상태 완료
- 계산서상태 완료
- 운송상태 완료

하나라도 미완료이면 종결할 수 없다.

예를 들어 거래상태, 출금상태, 계산서상태, 운송상태가 완료되었더라도 입금상태가 미완료이면 Formula는 종결되지 않는다.

미종결 Formula는 추적관리 대상이다.

대시보드와 관리 화면에서 미종결 사유를 노출해야 한다.

# 13. 실제 거래 검증 사례

## 13.1 포뮬러

```text
A.CJ제일제당
↓
B.지오웍스
↓
C.네이처인사이트
↓
D.에코앤리사이클
```

## 13.2 주요 특징

- 부가세 포함 거래
- 선입금 구조
- 세금계산서는 단계별 발행
- 실물은 A에서 D로 직송
- 지오웍스가 운송비를 부담하고 판매가에 운송비를 포함
- 매입 없는 회사 허용
- 매출 없는 최종 회사 허용

## 13.3 설계 반영점

- 매입가 0원 가능
- 매출가 0원 가능
- 최종 도착 회사는 매출/이익 확인 불가 가능
- 운송비 부담 주체와 운송사 정산은 별도 관리
- 세무 흐름과 실물 흐름은 분리 관리
