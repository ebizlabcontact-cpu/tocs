# 0. TOCS 핵심 인식

TOCS는 단순 거래관리, ERP, CRM, 그룹웨어가 아니다.

TOCS는 **Formula(포뮬러)** 를 중심으로 모든 운영 데이터를 발생시키고, 그 데이터를 파생·누적·취합하여 경영 판단 지표로 전환하는 **Trading Operation Control System** 이다.

TOCS의 핵심 데이터 흐름은 반드시 아래 구조를 따른다.

```text
Formula
  ↓
Derived Data
  ↓
Accumulated / Aggregated Data
  ↓
Dashboard
  ↓
KPI / Management Indicators
```

즉, TOCS의 모든 화면과 기능은 아래 질문에 답해야 한다.

```text
이 기능은 Formula에서 발생한 데이터를
정확하게 파생·누적·취합·시각화하는가?
```

이 구조와 직접 관련 없는 기능은 V1 범위에서 제외하거나 후순위로 둔다.

# 1. Formula First Architecture

## 1.1 Formula의 정의

Formula는 TOCS의 최상위 핵심 단위이자 원장(Source of Truth)이다.

Formula는 하나의 단순 거래가 아니다.  
Formula는 다자간 거래 구조, 참여 회사, 품목, 수량, 단가, 운송비, 입출금, 정산, 수익 구조를 포함하는 운영 원장이다.

## 1.2 Formula에 포함되는 핵심 데이터

Formula는 최소한 아래 데이터를 포함하거나 연결해야 한다.

- 참여 회사
- 회사별 역할
- 품목
- 수량
- 단가
- 매입 구조
- 매출 구조
- 운송사
- 운송비
- 부담 비용
- 입금 예정
- 출금 예정
- 실제 입금
- 실제 출금
- 미수금
- 미지급금
- 예상 순이익
- 확정 순이익
- 진행 상태
- 변경 이력
- 증빙 자료

# 2. TOCS 데이터 흐름

## 2.1 1단계: Formula 입력

실무자는 Formula 관리 화면에서 거래 구조를 입력한다.

입력 예시:

```text
A 공급사
↓
B 중개사
↓
C 유통사
↓
E 바이어
+
품목
수량
단가
운송비
입출금 조건
정산 조건
```

Formula 입력은 TOCS 전체 데이터의 출발점이다.

## 2.2 2단계: 파생 데이터 생성

Formula 입력값을 기준으로 시스템은 다음 데이터를 자동 계산한다.

- 매입총액
- 매출총액
- 운송비 총액
- 부담 비용
- 예상 순이익
- 확정 순이익
- 미수금
- 미지급금
- 거래처별 매입/매출 금액
- 운송사별 미지급 운임
- 품목별 거래 규모
- 기간별 거래 규모
- 단계별 정산 상태

## 2.3 3단계: 누적·취합

파생 데이터는 기간, 거래처, 품목, 운송사, 상태, 담당자 기준으로 누적·취합된다.

취합 기준 예시:

- 월별
- 분기별
- 연간
- 거래처별
- 품목별
- 운송사별
- 상태별
- 미수/미지급별
- 진행중/종결별

## 2.4 4단계: 대시보드화

누적·취합 데이터는 대시보드에서 경영 지표로 표현된다.

대시보드는 원본 입력 화면이 아니라, Formula에서 파생된 결과 데이터를 대표와 실무자가 빠르게 판단할 수 있게 보여주는 제어 화면이다.

## 2.5 5단계: KPI / 경영지표

TOCS KPI는 Formula 기반 파생 데이터로 산출한다.

대표 KPI 예시:

- 기간 내 총 거래액
- 기간 내 총 매입액
- 기간 내 총 매출액
- 예상 순이익
- 확정 순이익
- 미수금 총액
- 미지급금 총액
- 운송비 미지급액
- 진행중 Formula 수
- 종결 Formula 수
- 거래처별 수익 기여도
- 품목별 수익 기여도
- 월별 순이익 추이

# 3. Formula Integrity Rule

Formula는 시스템의 원장이다.

Formula 구조를 변경하면 파생 데이터, 누적 집계, 대시보드, KPI가 모두 영향을 받는다.

따라서 Formula 구조 변경은 최고 수준의 검토 대상이다.

새로운 엔티티, 기능, 화면, 테이블을 만들기 전 반드시 아래 질문을 수행한다.

```text
1. 이 데이터는 Formula에서 발생하는가?
2. 이 데이터는 Formula에서 파생 가능한가?
3. 이 기능은 Formula 데이터를 더 정확하게 관리하는가?
4. 이 기능은 대시보드/KPI 정확도를 높이는가?
```

위 질문에 답하지 못하는 기능은 V1 핵심 기능으로 채택하지 않는다.

---

# 4. 수평적 Entity 원칙

모든 회사는 수평적 Entity로 취급한다.

특정 회사를 시스템 중심에 두지 않는다.

우리 회사도 Formula 내 역할에 따라 다음 중 하나가 될 수 있다.

- 공급사
- 중개사
- 유통사
- 바이어
- 운송 관련 정산 대상

운송사 역시 독립 Entity로 관리한다.

회사 자체에 고정 역할을 박지 않는다.  
역할은 Formula 내부 참여 구조에서 결정한다.

---

# 5. Formula Version / Snapshot 원칙

Formula 변경 시 기존 데이터를 덮어쓰지 않는다.

중간 업체 변경, 단가 변경, 운송비 변경, 정산 조건 변경 등 핵심 정보가 바뀌면 신규 버전 또는 스냅샷을 생성한다.

기존 버전은 동결(Freeze)하여 과거 이력과 세무 증빙을 보존한다.

삭제보다 보존을 우선한다.

---

# 6. 핵심 기능 범위

V1 핵심 기능은 아래 4개다.

```text
1. Formula 관리
2. Formula 기반 정산 관리
3. Formula 기반 자금 관리
4. Formula 기반 대시보드/KPI
```

기타 기능은 위 4개 핵심 기능을 보조하는 범위에서만 허용한다.

---

# 7. 메뉴 구조 기준

권장 메뉴 구조:

```text
1. 통합 대시보드
2. Formula 관리
3. 자금/정산 관리
4. 품목 관리
5. 거래처/회사 관리
6. 시스템 설정
```

대시보드는 Formula 데이터를 입력하는 곳이 아니라, Formula에서 파생·누적·취합된 데이터를 확인하는 곳이다.

# 8. AI 개발 거버넌스

AI는 설계자가 아니다.  
AI는 설계 보조 도구다.

AI는 다음을 임의로 변경할 수 없다.

- Formula First Architecture
- Formula 중심 데이터 흐름
- 정산 방식
- 자금 흐름
- 권한 구조
- 핵심 Entity 구조

모든 AI는 본 문서를 기준으로 작업해야 한다.

# 10. 우선순위

TOCS 개발 우선순위는 다음과 같다.

2. 정산 정확성
